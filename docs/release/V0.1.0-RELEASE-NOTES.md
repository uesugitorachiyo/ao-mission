# AO Mission v0.1.0 Release Notes

AO Mission `v0.1.0` is the initial stable release of the local, durable mission
lifecycle and routing surface for the AO Stack.

## Included Capabilities

- Create, inspect, pause, resume, stop, archive, restore, and reconcile durable
  mission records across interrupted or multi-stage work.
- Route objectives through AO Blueprint, AO Atlas, AO Foundry, and AO Command
  using explicit continuation steps, checkpoints, return gates, and final
  rollups.
- Bind cross-component artifacts with correlation chains, SHA-256 manifests,
  readiness bundles, dashboards, event indexes, and verification bundles.
- Import and validate downstream workgraphs, scheduler readbacks, recovery
  evidence, factory rollups, and other versioned AO contracts.
- Replay Telegram, A2A, and scheduler fixtures and produce gateway readiness
  readbacks without contacting providers.
- Compact retained mission and timeline history while preserving digest-bound
  no-authority evidence.

## Supported Release Artifacts

The release publishes native archives for:

- Linux x86_64
- macOS aarch64
- Windows x86_64

Each archive is built on its native GitHub-hosted runner and contains the AO
Mission binary, license and notice files, exact version evidence, functional
smoke evidence, an SBOM, provenance, and the approved release manifest.

## Verification

Candidate archives are bound to the exact release source commit and approved
manifest. The release workflow verifies native binary identity, embedded
`ao-mission version=0.1.0` output, archive inventory, checksums, provenance,
functional contract validation, and an immutable promotion plan before the
protected publication job can run.

After extracting an archive, confirm the release identity with:

```sh
ao-mission --version
ao-mission doctor --json
```

## Safety Boundaries

AO Mission records intent, state, routing decisions, and evidence. It does not
execute downstream repository mutations, approve work, merge pull requests,
publish releases, deploy software, or grant provider authority. Gateway and
scheduler surfaces remain intent/readback-only; fixture replay does not contact
Telegram, A2A peers, schedulers, or model providers. Telegram token values are
read from the operator-selected environment variable and are not printed or
persisted.

The local A2A fixture surface does not provide arbitrary tools, streaming, push
notifications, or mutation authority. Imported evidence is validated and
recorded as readback; it does not inherit execution or approval authority from
its producer. RSI remains denied.
